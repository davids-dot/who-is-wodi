## Context

盒子环境提供 cube-llm 服务（Qwen3.6-35B-A3B 模型，OpenAI 兼容 API），支持文本对话和流式响应。FDE 模板已内置 `server/llm-client.js` 封装了 chat/vision/embeddings/rerank 能力。本项目利用该 LLM 能力构建一个"谁是卧底"AI 对战游戏，6 个虚拟人物由 LLM 驱动，用户作为旁观者观看游戏进程。

技术栈约束：React 18 + TypeScript + Vite + Ant Design v6 + Less + CSS Modules，Node.js 后端，appKey 为 `who-is-wodi`。

## Goals / Non-Goals

**Goals:**
- 实现完整的"谁是卧底"游戏流程：发牌→描述→投票→淘汰→胜负判定
- 6 个 AI 玩家具不同人设，依次描述，描述内容不重复
- "禁字"规则作为游戏规则告知模型（非硬性约束），投票时作为可疑线索
- 卧底不知道自己是卧底（prompt 结构完全相同，仅词语不同）
- SSE 流式推送描述内容，前端打字机效果
- 圆桌 UI 布局，历史记录可查看
- 用户控制游戏节奏（开始/下一轮）

**Non-Goals:**
- 不支持真人参与游戏（用户仅旁观）
- 不实现白板角色（5+1 结构足够）
- 不持久化游戏数据（内存存储，重启重置）
- 不实现多房间/多人在线功能
- 不对描述做硬性禁字过滤（规则告知模型，违反由投票环节自然处理）

## Decisions

### 1. 游戏状态机设计

**决策**：在 Node.js 后端用内存对象管理游戏状态，状态机包含 6 个状态。

```
IDLE → DEALING → DESCRIBING → VOTING → RESULT → (GAME_OVER | 等待下一轮 → DESCRIBING)
```

**理由**：无需数据库，游戏状态在进程内存中即可。单用户旁观场景不需要多实例共享状态。

**替代方案**：用 Redis 持久化 → 过重，单局游戏不需要持久化。

### 2. LLM 调用策略

**决策**：
- **描述阶段**：依次调用（P1→P6），每个玩家能看到之前所有玩家的描述。使用流式 `chat({stream: true})`，通过 SSE 逐字推送到前端。
- **投票阶段**：并行调用 6 次 `chat()`（非流式），每个玩家独立分析后返回 JSON 格式投票结果。

**理由**：描述需要依次进行（后描述者能看到前者的描述），投票可以并行（所有人都基于相同的完整描述集投票）。

### 3. 卧底信息隔离机制

**决策**：卧底和平民的 prompt 结构 100% 相同，唯一区别是 `{word}` 的值。系统在服务端标记卧底身份，但 prompt 中不包含"你是卧底"或"你是平民"的信息。

```javascript
// 所有玩家统一的 prompt 模板
const prompt = {
  systemPrompt: `你正在参与"谁是卧底"游戏...`,
  messages: [{
    role: 'user',
    content: `你的词语是：${player.word}\n...`
  }]
}
// player.word 对于平民是 civilianWord，对于卧底是 undercoverWord
// 但 prompt 结构完全相同，玩家无法从 prompt 结构判断自己的身份
```

**理由**：这是游戏核心机制。如果 prompt 中告知身份，卧底会刻意伪装，失去"一本正经胡说八道"的喜剧效果。

### 4. 禁字规则作为"软规则"

**决策**：在 prompt 中告知模型禁字规则（"描述中不能包含你词语中的任何一个字"），但服务端不做硬性过滤。投票 prompt 中引导模型关注"谁可能违反了禁字规则"。

**理由**：
- 模型偶尔违反禁字规则正是游戏的趣味所在——卧底可能无意中使用平民的禁字而暴露自己
- 如果服务端硬性过滤并重试，会消除这种自然的"泄密"机制
- 投票时"谁用了不该用的字"成为重要的推理线索

### 5. 前端圆桌布局

**决策**：使用 CSS 绝对定位实现 6 个座位的环形布局，不使用 Canvas/SVG。

```
       P1(顶部)
   P6        P2
   P5        P3
       P4(底部)
```

**理由**：CSS 定位足够实现圆桌效果，且响应式适配更简单。Canvas/SVG 增加复杂度且交互处理麻烦。

### 6. SSE 推送协议

**决策**：`/api/who-is-wodi/game/next-round` 返回 SSE 流，事件格式：

```
event: describe_start   data: {"playerId":1,"playerName":"老王"}
event: describe_chunk   data: {"playerId":1,"text":"甜"}
event: describe_chunk   data: {"playerId":1,"text":"甜的"}
event: describe_end     data: {"playerId":1,"fullText":"甜甜的，下午来一杯很幸福"}
event: round_complete   data: {"round":1}
```

投票结果通过单独的 POST `/api/who-is-wodi/game/vote` 返回 JSON。

**理由**：描述阶段需要打字机效果，SSE 最适合。投票结果是一次性数据，用普通 JSON 响应即可。

### 7. 词库筛选标准

**决策**：词对筛选需满足：
- 两个词语义相近但不同（确保卧底描述方向有偏差）
- 每个词的组成字不太常见（避免禁字过于严苛导致玩家无法正常说话）
- 避免包含"子、的、大、小"等极高频字的词

**示例**：`奶茶/豆浆`（禁字：奶茶/豆浆，都不太常见）✅ ；`橘子/柚子`（禁字都含"子"）❌

## Risks / Trade-offs

- **[LLM 响应延迟]** 每轮 6 次串行 LLM 调用，加上流式传输，单轮可能需要 30-60 秒 → 前端展示加载动画和"正在思考..."提示，保持用户耐心
- **[LLM 输出不稳定]** 模型可能偶尔输出不符合格式的内容（如带引号、超长） → 服务端做轻量后处理（trim、截断），不做重试
- **[禁字规则遵守率]** 模型可能不完美遵守禁字规则 → 这正是设计意图，违反规则成为投票线索，不是 bug 而是 feature
- **[投票 JSON 解析失败]** 模型可能返回非 JSON 格式的投票 → 服务端做 fallback：解析失败时随机投票并标注"直觉投票"
- **[内存状态丢失]** 服务重启后游戏状态丢失 → 可接受，游戏为单局娱乐性质，用户重新开始即可
- **[词库质量]** 词对如果太相似或太不同都会影响游戏体验 → 精心筛选 50+ 组，确保相似度适中
