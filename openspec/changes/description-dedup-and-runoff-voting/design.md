## Context

当前游戏引擎在 `engine.js` 中处理描述生成和投票淘汰逻辑：

1. **描述生成**：`buildDescriptionUserMessage` 构建 prompt，传递历史描述和当前轮次描述，但 prompt 中没有明确强调"不能重复自己之前说过的描述"

2. **投票淘汰**：`eliminatePlayer` 函数统计票数，当最高票有多个时直接判定为平票（`isTie = true`），无人淘汰

## Goals / Non-Goals

**Goals:**
- 确保所有玩家在所有轮次的描述全局唯一（跨玩家、跨轮次）
- 实现决胜投票机制，平票时由非最高票玩家进行第二轮投票
- 保持前端兼容，决胜投票对前端透明

**Non-Goals:**
- 不改变描述生成的其他规则（字数限制、语气风格等）
- 不修改前端界面
- 不引入新的游戏状态（决胜投票复用现有投票流程）

## Decisions

### 1. 描述去重实现方式

**选择**：强化 prompt，明确列出所有历史描述并强调不能重复

**理由**：
- 无需修改数据结构，利用现有的 `history` 和 `currentDescriptions` 参数
- LLM 有能力理解并遵守"不能重复"的约束
- 实现简单，风险低

**Prompt 修改**：
```
历史描述记录（你不能重复以下任何描述，包括你自己之前说过的）：
第1轮：
  老王: "夏日的噪音"
  小美: "树上叫个不停"
...

要求：
1. 不能重复以上任何历史描述
2. 15个字以内
3. ...
```

### 2. 决胜投票流程设计

**选择**：内嵌在 `eliminatePlayer` 中，递归或循环调用直到有结果

**流程**：
```
eliminatePlayer(gameId, votes, isRunoff = false):
  1. 统计票数
  2. 找出最高票集合 M
  3. 如果 M.size === 1 → 淘汰该玩家，返回结果
  4. 如果 M.size > 1:
     a. 找出非最高票玩家集合 N
     b. 如果 N 为空 → 平票，返回结果
     c. 让 N 中的玩家对 M 中的候选人重新投票
     d. 递归调用 eliminatePlayer 处理新票数
```

**理由**：
- 复用现有投票逻辑，无需新增游戏状态
- 对前端透明，前端只需等待最终结果
- 实现简洁，易于测试

### 3. 决胜投票的 prompt 调整

**选择**：复用现有 `buildVoteUserMessage`，但限制候选人为 M

**理由**：
- 保持投票逻辑一致性
- 只需在决胜轮传入不同的候选人列表

## Risks / Trade-offs

| Risk | Mitigation |
|------|-----------|
| LLM 仍可能重复描述 | 在 prompt 中明确列出所有历史描述，强化约束 |
| 决胜轮可能再次平票 | 递归处理，如果再次平票且无法继续决胜，则判定为平票 |
| 决胜轮增加游戏时间 | 决胜轮通常能快速决出结果，最多增加一轮投票时间 |
| 递归深度问题 | 决胜轮最多进行 2-3 轮（人数递减），不会出现无限递归 |

## Open Questions

1. 决胜投票时，AI 的投票理由是否需要特别说明是"决胜轮投票"？
   - 建议：不需要，保持理由自然

2. 如果决胜轮后仍平票（如 2v2 变成 1v1），是否继续决胜？
   - 建议：如果非最高票玩家集合为空，则停止决胜，判定平票
